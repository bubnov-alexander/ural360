# План миграции

## Цель

Постепенно перенести Ural360 с WordPress-инфраструктуры на Laravel Apiato, сохранив рабочий Telegram-бот и не ломая текущий процесс приема заявок.

## Этап 1. Инвентаризация legacy

- Зафиксировать WordPress-страницы, услуги, галерею и WooCommerce-сущности.
- Описать текущие сценарии Telegram-бота.
- Выделить сущности, которые должны стать общими для сайта, админки и бота.
- Определить, какие данные являются справочниками, а какие операционными заявками.

## Этап 2. Проектирование Apiato Containers

Предварительные контейнеры:

| Container | Ответственность |
| --- | --- |
| `Service` | Услуги: катамараны, SUP, трансфер, автостоянка |
| `Route` | Туристические маршруты и сплавы |
| `Booking` | Заявки, бронирования, статусы, история обработки |
| `Customer` | Клиенты и контактные данные |
| `Media` | Галереи и файлы через Spatie Media Library |
| `TelegramBot` | Интеграционный слой для бота |
| `Admin` | Filament-ресурсы и административные сценарии |

## Этап 3. Filament admin

- Создать ресурсы для услуг и маршрутов.
- Добавить управление заявками и статусами.
- Подключить загрузку изображений через Spatie Media Library.
- Ограничить доступ к панели через роль администратора.
- Сделать seeder для первого пользователя админки через `.env`.

## Этап 4. Общая база для бота

На первом этапе бот может работать с общей БД, но только через отдельного пользователя.

Разрешенные операции должны быть минимальными:

- чтение справочников услуг;
- создание заявок;
- обновление только своих рабочих полей заявки;
- чтение статусов, которые нужны для сценария бота.

Запрещено:

- изменять структуру БД;
- выполнять migrations;
- читать таблицы пользователей админки и системные таблицы;
- иметь полный доступ ко всей базе.

## Этап 5. Импорт WordPress

- Перенести услуги и маршруты в новые таблицы.
- Перенести галерею и изображения в Spatie Media Library.
- Отдельно обработать SEO-данные, slug и публичные URL.
- Проверить, какие WooCommerce-сущности реально нужны в новой модели.

## Этап 6. Переход Telegram-бота на API

После стабилизации backend-логики бот должен перейти с прямого доступа к БД на Apiato API.

Плюсы:

- вся валидация в одном месте;
- проще менять структуру БД;
- меньше риска дать боту лишние права;
- можно логировать действия через backend;
- легче подключать другие клиенты.

## Этап 7. Замена WordPress

Финальный этап - переключить публичный сайт на новую платформу. WordPress останется только как источник данных для миграции или будет полностью выведен из эксплуатации.
