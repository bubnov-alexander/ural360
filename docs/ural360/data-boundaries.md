# Границы данных и доступ Telegram-бота

## Принцип

Общая база данных допустима как переходное решение, но владельцем схемы должен быть Laravel/Apiato backend. Telegram-бот не должен управлять миграциями, создавать таблицы или иметь полный доступ к данным.

## Роли доступа

| Роль | Назначение |
| --- | --- |
| Apiato application user | Полный прикладной доступ, migrations, бизнес-логика |
| Filament admin user | Управление через админку согласно ролям |
| Telegram bot DB user | Минимальный доступ к разрешенным таблицам |
| Readonly/report user | Только чтение для отчетов, если понадобится |

## Что можно дать боту

- `SELECT` по справочникам услуг и маршрутов;
- `INSERT` в таблицу заявок;
- `UPDATE` по ограниченному набору полей заявок;
- `SELECT` по статусам, которые нужны для сценария бота.

## Что нельзя давать боту

- `DROP`, `ALTER`, `CREATE`, `TRUNCATE`;
- доступ к таблицам пользователей админки;
- доступ к токенам, настройкам, системным таблицам и логам;
- право менять роли и права;
- полный доступ к персональным данным вне своего сценария.

## Почему это важно

Telegram-бот является внешним процессом относительно backend-приложения. Если он получит полный доступ к БД, любая ошибка в боте станет ошибкой всего проекта. Ограниченный DB-пользователь снижает риск и позволяет спокойно мигрировать систему по частям.

## Целевое состояние

Прямой доступ бота к БД должен быть временным. В целевой архитектуре бот вызывает Apiato API, а backend сам решает:

- какие данные вернуть;
- какие поля можно изменить;
- какие статусы допустимы;
- какие события надо залогировать;
- какие уведомления отправить дальше.
